# Push-based continuous deployment via GHCR

**Date:** 2026-07-07 · **Context:** with the M4 public-deploy prep landed (RUNBOOK §4 public-TLS path, `scripts/prod-smoke.sh`, the deploy checklist), the remaining ergonomic gap is *ongoing* releases: the manual "SSH in → `git pull` → `docker compose up -d --build`" is slow, builds on the 2 GB box, and is easy to do half-way. This ADR records the CD design chosen with the owner (who wanted a "typical CI deploy — merge to main and it ships"). Engineering/doc prep only — no VPS/secrets exist yet; everything account-gated is queued in `docs/DEPLOY_CHECKLIST.md` Gate 5.

## Decision

**Push-based CD: merge to `main` → GitHub Actions builds the image and pushes it to GHCR → Actions SSHes into the VPS and runs an on-box deploy script.** Implemented as `.github/workflows/deploy.yml` + `compose.prod.yaml` + `scripts/deploy.sh` + `.github/dependabot.yml`.

1. **Build off the box, ship an immutable digest.** The single image (`Dockerfile.api` — SPA baked in, served same-origin) is built in CI and pushed to `ghcr.io/braedensc/chefclaw` (`sha-<short>` + `latest` for humans). The box no longer builds — it **pulls the exact `@sha256` digest** CI produced, handed job→job, so what ran through CI is byte-identical to what serves. `compose.prod.yaml` pins both `migrate` and `api` (one shared image) to `${CHEFCLAW_IMAGE}` with `pull_policy: always`; the base `build:` block stays inert because the deploy never passes `--build`.

2. **Two-job trust split (public-repo containment).** The `build` job runs untrusted code (deps, Dockerfile) and holds the *only* write scope (`packages: write`) but **no SSH key**; the `deploy` job is trusted, holds the SSH key, and has `contents: read` only — no registry scope (the box pulls the public image anonymously). Top-level `permissions: {}` denies all; each job re-grants the minimum. Third-party actions are pinned to 40-hex commit SHAs (Dependabot-maintained). The workflow triggers on `push` to `main` (+ `workflow_dispatch`) and **never** `pull_request[_target]`, so fork code can't reach the deploy secrets.

3. **The SSH key can only deploy.** The box's `authorized_keys` pins the deploy key to a **forced command** (`command="/opt/chefclaw/scripts/deploy.sh",restrict`), so a leaked key gets no shell — only a deploy. CI passes the target digest as `$SSH_ORIGINAL_COMMAND`; `deploy.sh` strict-allowlists it (`^ghcr\.io/braedensc/chefclaw@sha256:[0-9a-f]{64}$`), never eval's it. Host key is pinned (`DEPLOY_KNOWN_HOSTS`, `StrictHostKeyChecking=yes`) — no trust-on-first-use.

4. **On-box safety rails (`deploy.sh`).** In order: capture the running image as a *pullable* rollback ref → **backup first** (`scripts/backup.sh`) → `git merge --ff-only` (abort on a diverged box tree) → `docker compose pull` → **migrate and gate on its exit code** (a bad migration aborts *before* the api swap; old api keeps serving) → `up -d --no-build --no-deps api` → **health-gate** on `GET http://127.0.0.1:8000/` (the unauthenticated SPA index; `/api/health` is 401 by design and is never probed) → **auto-rollback** to the previous image (`--pull never`) and exit non-zero on failure. The SSH exit status *is* the deploy result, so a failed migration/health-gate reds the Actions job.

5. **`CD_ENABLED` master switch.** The `deploy` job runs only when the repo variable `CD_ENABLED == 'true'`; otherwise it **skips** (grey, not red). This keeps the pipeline-introducing PR from firing a half-configured deploy, and is the clean pause/resume lever. The `build` job always runs (a fresh GHCR image per merge is harmless).

## Why

- **Push-based over pull-based** was the owner's explicit call — it's the familiar "CI deploys my server" shape, with deploy logs in the Actions UI. The cost (a deploy credential in a public repo's CI) is bounded by the forced command, the two-job trust split, the `main`-only/never-fork triggers, and the `production` Environment scoping the secrets. Pull-based (a box-side GHCR poller, no CI creds) was the safer-by-default alternative and is recorded as the rejected option; it's a small change if the credential surface ever feels wrong.
- **Build in CI, pull a digest** moves the heavy SPA+image build off the 2 GB box (where it contends with the running app and leans on swap) and makes rollback deterministic — re-run an older digest.
- **Human-gated, not fully unattended.** The health-gate + auto-rollback make a bad *code* deploy self-correcting, but the design deliberately keeps a human in the loop for the parts that matter: migrations run before the swap and are **not** auto-reverted (rollback reverts code, never schema — keep migrations expand-then-contract/backward-compatible; a bad one needs a DB restore, RUNBOOK §2), and the first deploy is attended (no prior image to roll back to). This fits the single-worker/no-broker, single-owner, irreplaceable-DB reality.
- **No zero-downtime attempt.** True zero-downtime needs the deferred TaskIQ multi-worker rebuild + a load balancer + expand/contract discipline, and would fight the concurrency-1 double-spend invariant. A few seconds of downtime per release is a non-issue for an invite-only app; engineering it away here would be effort against a problem that doesn't exist.

**Alternatives rejected by name:** pull-based poller / Watchtower (safer credential model, but Watchtower can't run migrations or health-gate the swap; poller was the owner's second choice); full auto-deploy with unattended migrations (unsafe for one-way migrations + a single owner); a PaaS (Render/Railway/Fly — "easier" deploys, but re-architects storage to managed PG + object storage, must pin to one instance to preserve the worker invariant, costs more, and discards the M-Deploy/M4 compose-on-VPS prep); building on the box (slow, RAM-contended).

## Verified

Prep, not a live run. The pipeline was designed and adversarially reviewed by a multi-agent workflow (recon against the real `compose.yaml`/`Dockerfile.api`/`app.py`/`backup.sh`, a security + correctness + ops review pass); its blockers were fixed in the committed artifacts (pullable rollback ref — not the bare `{{.Image}}` config id; `--pull never` on rollback; docker-socket/`git`-ownership preflight; `--no-deps` to avoid a double migrate). `scripts/deploy.sh` passes `dash -n`/`sh -n` and `shellcheck`; `deploy.yml` is valid workflow YAML with the action SHAs resolved against live tags on 2026-07-07. **First real exercise is at deploy day** (checklist Gate 5) — the SSH forced command, GHCR anonymous pull, and the full backup→migrate→health→rollback chain have never run against a real box. Known residual limits, all documented in-script and in RUNBOOK §4: the health-gate is **liveness-only** (SPA 200 ≠ DB/worker healthy — the future unauthenticated `/api/livez` readiness probe is the fix, its own ADR); rollback ≠ schema revert; the first deploy is un-rollback-able and attended; the GHCR-package-public step is order-sensitive.
