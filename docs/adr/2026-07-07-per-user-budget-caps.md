# M3: per-user budget/rate caps + paid Gemini tier

**Date:** 2026-07-07 · **Context:** milestone M3 (branch `feat/m3-per-user-caps`), the third phase of the Path-B multi-user product (`docs/adr/2026-07-07-path-b-multi-user-product.md`). Builds on the M2 readiness columns and the V2-A spend design (`docs/adr/2026-07-06-observability-and-cost.md`).

## Decision

Wire the two M2-readiness columns (`users.monthly_budget_usd`, `users.max_attempts_per_day`, both nullable) into the paid-call gate, expose an admin write for them, and surface the effective cap to the operator:

1. **Per-user caps override the global default in `check_budget`.** The gate still parses the global `MONTHLY_LLM_BUDGET_USD` / `MAX_EXTRACTION_ATTEMPTS_PER_DAY` **first** (fail-closed: unset/unparseable ⇒ `ConfigError`, no paid calls — a per-user cap can NEVER re-enable spend the operator has not globally enabled). Then, in the **same session** as the ledger reads, it reads the owner's two columns; a non-NULL column replaces the global value (higher or lower), NULL falls back to it. The read sits between the config parse and the ledger reads so it shares the gate's read snapshot — the double-spend gate is preserved (still safe only at worker concurrency 1; `docs/adr/2026-07-06-jobs-without-broker.md`).
2. **`PATCH /api/admin/users/{user_id}/budget`** behind transport-layer `require_admin` (like the invite admin routes) sets either cap. Partial-update semantics: a field present sets the override, present-and-`null` clears it back to the global default, absent leaves it unchanged; positive values only (a `Numeric(10,2)` monthly, an integer daily). No migration — the columns already exist.
3. **Health surfaces the EFFECTIVE cap.** `/api/health` (owner-scoped) now reports the per-user-or-global monthly budget and daily cap in the existing `budget_monthly_usd`/`daily_attempt_cap` fields plus a `budget_is_personal` flag; the Settings spend-bar already tracks those fields, so it now tracks the caller's real cap, and labels it a personal cap when overridden. Fail-closed ⇒ still `null`; a failed per-user read degrades to the global cap (never raises).
4. **Paid Gemini tier is selectable globally AND per-user.** `GEMINI_MODEL` (`gemini-2.5-flash` default = free) is the global default everyone runs; `GEMINI_PAID_MODEL` (`gemini-2.5-pro` default) is the paid model. A new `users.paid_tier` boolean (default false; set via the same admin budget endpoint) bumps THAT account's extractions to the paid model — the single swap lives in `extractors.extractor_settings_for_tier(settings, paid_tier=...)`, used by both the worker (extractor construction + ledger attribution) and `/api/health` (the caller's effective `model`), so they can never disagree. The `GEMINI_PRICING` table already prices pro (padded), so the fail-closed budget gate bounds pro spend unchanged — a paid-tier owner just hits their cap faster. **The per-user read is separate from the double-spend gate** (model choice is not a spend-safety invariant), so it rides its own short store call, not the gate transaction.

## Why

- **Global master switch, per-user tuning.** Keeping the global parse first means the operator's fail-closed switch is unforgeable — a per-user column can only redistribute or tighten within an already-enabled budget, never open a leak. Owner-scoped spend (`month_to_date_usd`/`attempts_today` already filter by `owner_id`) makes the "global" env value function as a per-user *default*, so a per-user override replacing it (either direction) is coherent, not a second pool.
- **Same-transaction read over a cached read.** Reading the caps in the gate's own session (not at enqueue, not from a request-time cache) keeps the check honest under the pre-call retry path and leaves the concurrency-1 double-spend argument intact. Rejected: threading a caps object down from the router (goes stale across retries and requeues) and a separate pre-read session (a second snapshot the gate can't reason about).
- **Partial PATCH via `model_fields_set`** distinguishes "clear to global" (`null`) from "leave alone" (absent) for two independent columns — the REST-correct semantic — instead of forcing the admin to re-send both every time.
- **Per-user paid tier, requested (2026-07-07, Braeden).** The first cut shipped the global flip only and flagged per-user model selection as deferred; Braeden asked to enable it, so `users.paid_tier` + `GEMINI_PAID_MODEL` land in the same PR. Kept as a **boolean flag mapping to a configured model id**, NOT a free-form per-user model string — the admin can't fat-finger a model, and cost stays bounded by the (padded) pricing table regardless. The global `GEMINI_MODEL` stays the baseline everyone gets; `paid_tier` only *upgrades* an account to `GEMINI_PAID_MODEL`. Scoped to the EXTRACTION model only (image/illustration generation stays global — it isn't the expensive call, and per-user cover models weren't asked for). Rides its own short store read, not the gate transaction: model choice is not a spend-safety invariant, so it needn't share the double-spend snapshot.
- **The caps read needs no migration; the paid-tier flag does.** The M2 caps columns already existed, so the caps work is pure logic+transport. `paid_tier` is a new column ⇒ one additive migration (`f3a4b5c6d7e8`, Boolean NOT NULL default false — every existing row backfills to free, no data migration), chained off the current head after a fresh `main` pull per the serialization rule.

## Verified

- **Unit tier (no DB, `uv run pytest -q`, 415 tests):** `check_budget` with a monkeypatched `read_user_caps` — a per-user monthly/daily override gates independently of the global; NULL columns fall back to the global; a per-user cap does **not** bypass a fail-closed global (`ConfigError` still raised before any read); config-error-before-any-ledger-read order preserved. Worker: a paid-tier owner's extractor is built with `gemini-2.5-pro` and a free-tier owner's with `gemini-2.5-flash` (recording factory); a paid-tier attempt that dies pre-usage still ledgers the paid model. Admin endpoint: 200 set (caps + `paid_tier`), 403 non-admin (re-stubbed `fetch_account`), 404 missing user, partial update + clear-to-null semantics, non-positive/extra-field rejected 422. Health: effective per-user cap surfaces with `budget_is_personal=true`; a paid-tier owner's `model` shows the paid model with `paid_tier=true`; free-tier shows the global model; fail-closed shows `null`.
- **Golden DB tier (`-m golden`, throwaway PG, 38 tests):** two users with different non-NULL caps gated independently against a real ledger + real `users` rows; a NULL-column user falls back to the global; the admin write round-trips through the real columns (`Numeric(10,2)` quantization; `paid_tier` set/read); `alembic upgrade head` applies `f3a4b5c6d7e8` and a new user backfills `paid_tier=false`.
- **Contract:** `backend/openapi.json` + `frontend/src/client` regenerated and committed (deterministic-verified); `ruff`, `pytest -q`, `npm run typecheck`/`lint`, 110 vitest green; CI OpenAPI-drift job passes.
